# 英语阅读功能（移动端优先）设计方案

## 1. 背景与目标

当前系统在手机网页上的 UI 兼容性较弱，且缺少系统化的英语阅读能力。目标是构建一个“可持续输入 + 可追踪输出 + AI 精读”的闭环，帮助用户稳定提升英语阅读能力。

核心目标：

1. **文章自动获取**：尽量避免人工导入，保证内容持续更新。
2. **阅读过程可标注**：支持记录单词、句子与笔记。
3. **AI 精读辅助**：对文章进行分段讲解、语法分析、难句拆解与练习。
4. **移动端优先体验**：页面布局、交互和性能先服务手机场景。

---

## 2. 用户场景与关键流程

### 2.1 典型用户流程

1. 打开移动端首页，看到“今日推荐文章”。
2. 进入阅读页，按段落滚动阅读，点击单词可查看释义。
3. 长按句子可“加入精读清单”或“收藏到复习卡片”。
4. 阅读后进入 AI 精读页，查看结构解析、难句讲解和练习题。
5. 系统自动把生词和句子加入复习计划（间隔复习）。

### 2.2 关键数据流

- 内容流：抓取源 -> 清洗 -> 分级 -> 入库 -> 推荐。
- 学习流：阅读行为 -> 标注/收藏 -> AI 分析 -> 复习计划 -> 学习进度。

---

## 3. 文章来源设计（自动拉取）

## 3.1 内容来源策略（优先级）

1. **可授权/开放 API 内容源（优先）**  
   例如新闻 API、开放出版平台、教育类可二次使用内容。
2. **RSS/Atom 源**  
   通过订阅源定时拉取新文章，成本低、稳定性高。
3. **站点采集（需合规）**  
   仅抓取允许抓取与使用的站点，遵守 robots.txt 和版权协议。

> 建议先从 API + RSS 启动，避免一开始就投入复杂爬虫治理。

### 3.2 内容处理流水线

1. **拉取（Scheduler）**：按来源定时任务（如每 30~60 分钟）。
2. **去重（Fingerprint）**：按 URL + 文本哈希防重复。
3. **清洗（Cleaner）**：去广告、去导航、正文提取、分段。
4. **语言与难度分级（NLP）**：词汇密度、句长、CEFR 粗分（A2~C1）。
5. **安全与合规（Moderation + License Check）**。
6. **入库与索引（DB + Search）**：支持标签、难度、主题检索。

### 3.3 数据模型（简版）

- `articles`: id, title, source, author, published_at, license, level, topics, content, summary
- `article_segments`: article_id, segment_index, text, vocab_density
- `ingestion_jobs`: source_id, status, pulled_at, error_log

## 4. 词句记录功能设计（网页版可实现）

## 4.1 能力定义

1. **单词记录**：点击词汇 -> 弹层展示词性、释义、例句、发音 -> 一键收藏。
2. **句子记录**：长按句子/划选文本 -> 收藏到“句库”。
3. **阅读笔记**：对段落添加个人理解、翻译或疑问。
4. **标签体系**：如 `生词`、`难句`、`语法`、`表达`。

### 4.2 Web 技术可行性

移动 Web 可以实现：

- 文字选择监听：`Selection API`
- 单词点击定位：分词后 token 包裹 + 事件代理
- 离线临时保存：`IndexedDB` / `localStorage`
- 云端同步：登录后批量上报收藏与笔记

### 4.3 相关数据模型

- `user_vocab`: user_id, lemma, word_form, context_sentence, article_id, first_seen_at, review_state
- `user_sentences`: user_id, sentence, article_id, note, tags, created_at
- `user_notes`: user_id, article_id, segment_index, content, created_at

## 5. AI 精读功能设计

## 5.1 AI 精读输出结构（建议固定模板）

对每篇文章输出 5 个模块：

1. **文章主旨**：3~5 行摘要 + 作者意图。
2. **段落导读**：每段一句“读这一段要抓什么”。
3. **难句拆解**：句法树（简化）、从句关系、替换表达。
4. **词汇拓展**：高频词组、近义替换、常见搭配。
5. **练习任务**：选择题、改写题、翻译题、复述提示。

### 5.2 AI 交互模式

- **自动模式**：用户读完后自动生成精读报告。
- **问答模式**：用户可追问“这句为什么用虚拟语气？”
- **对比模式**：对比用户翻译与参考译文，指出偏差。

### 5.3 成本与性能控制

- 先做分段摘要再合并，减少一次性长上下文开销。
- 缓存 AI 结果（按 article_id + model_version）。
- 对高频文章先离线预生成精读结果。

## 6. 移动端 UI/UX 重构建议（重点）

## 6.1 设计原则

1. **Mobile First**：先做 375px 宽度，再向上适配平板与桌面。
2. **单手操作优先**：核心按钮放在底部拇指热区。
3. **减少跳转**：阅读、标注、查词、收藏尽量在同页完成。
4. **可读性优先**：字号、行高、段间距可调。

### 6.2 页面结构建议

1. **首页**：今日推荐、按难度入口、最近学习进度。
2. **阅读页**：正文 + 悬浮工具条（查词/收藏/精读）。
3. **精读页**：卡片化展示 AI 解析与练习。
4. **复习页**：单词卡、句子卡、错题回顾。

### 6.3 组件级改造点

- 统一 `Bottom Sheet` 作为移动端弹层。
- 统一 `Sticky Bottom Action Bar` 放核心动作。
- 减少多列布局，改成纵向卡片流。
- 支持暗黑模式与护眼主题。

### 6.4 性能要求

- 首屏关键内容优先渲染。
- 文章正文按段懒加载（超长文本场景）。
- 图片延迟加载，减少移动网络流量。

## 7. 推荐与学习闭环

## 7.1 推荐逻辑（初版）

- 输入：用户词汇掌握度、阅读时长、历史难度、兴趣主题。
- 输出：70% 可理解输入 + 20% 略挑战 + 10% 高挑战。

### 7.2 复习机制

- 词句进入 SRS（间隔复习）队列。
- 每日推送“10 个生词 + 3 个难句复习”。
- 根据回忆成功率自动调整复习间隔。

---

## 8. 分阶段落地计划

## Phase 1（2~3 周）：MVP

- 移动端阅读页重构（可读性 + 选词/划句收藏）
- RSS/API 自动拉取 + 入库 + 基础推荐
- AI 精读基础版（主旨 + 难句 + 词汇拓展）

## Phase 2（3~5 周）：增强

- 精读问答模式
- 复习页（SRS）
- 难度分级与个性化推荐优化

## Phase 3（持续）：运营化

- 学习周报
- 学习目标体系
- A/B 测试（阅读完成率、复习留存）

---

## 9. 风险与应对

1. **版权风险**：仅接入可授权来源；保留来源、链接、许可证字段。
2. **AI 幻觉风险**：关键语法点提供“解释依据”，增加低温输出与审核阈值。
3. **移动端复杂交互稳定性**：优先实现核心手势，避免过度手势冲突。
4. **成本风险**：缓存 + 预生成 + 分层模型调用。

---

## 10. 建议的下一步（可直接执行）

1. 先确定 3~5 个合规内容源（API/RSS）。
2. 先改阅读页移动端框架与交互（这会直接影响后续功能可用性）。
3. 同步定义 `articles / user_vocab / user_sentences` 三张核心表。
4. 上线 MVP 后追踪三个指标：阅读完成率、收藏转化率、7 日复习留存。
